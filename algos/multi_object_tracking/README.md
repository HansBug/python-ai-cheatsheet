## 这是什么？

多目标跟踪（Multi-Object Tracking, MOT）解决的问题不是“这一帧里有什么目标”，而是：

> 连续多帧里，哪些检测框其实是同一个目标，它们的 ID 应该怎么稳定延续下去。

自动驾驶里它为什么重要，很好理解：

- 检测只能告诉你“这一帧这里有车、有行人”
- 跟踪要进一步告诉你“这辆车就是上一帧那辆车，它正在向左变道”

所以检测更像“逐帧识别”，跟踪更像“跨帧身份管理 + 运动估计”。

## 核心机制

### 1. 多目标跟踪的主链路是什么？

最常见的 online MOT 主线是：

```text
detector -> motion prediction -> data association -> track update -> lifecycle management
```

翻成中文就是：

1. 先用检测器拿到当前帧候选框
2. 用运动模型预测上一帧目标现在大概会在哪里
3. 把预测轨迹和当前检测做匹配
4. 匹配上的轨迹更新状态
5. 没匹配上的轨迹或检测分别做丢失 / 新建处理

这条链你只要能讲顺，自动驾驶感知岗面试里就已经能撑住一大半 MOT 基础题。

### 2. 为什么跟踪不能只靠检测？

因为检测器是逐帧工作的，它不关心“上一帧的车”和“这一帧的车”是不是同一个。

举个很直观的例子：

- 帧 1 里，左边有一辆白车，右边有一辆黑车
- 帧 2 里，两辆车都往前开了一点
- 如果只看检测框，你只能说“当前有两个框”
- 如果做跟踪，你才知道白车的 ID 是不是还延续为 `track_3`

自动驾驶里后续的速度估计、轨迹预测、风险判断都要依赖这种连续身份。

### 3. 关联（association）到底在干什么？

关联的本质是：

> 当前帧每个检测框，应该匹配给哪条历史轨迹？

最常见的做法，是先定义“相似度”或“代价”，再做最优匹配。

常见相似度包括：

- IoU
- 边框中心距离
- 马氏距离
- ReID 特征相似度

如果先讲最简单的版本，IoU 已经够用了：

$$ \mathrm{IoU}(A, B) = \frac{|A \cap B|}{|A \cup B|} $$

这里：

- $A, B$：两个边框
- $|A \cap B|$：交集面积
- $|A \cup B|$：并集面积

如果两个框重叠越多，说明它们越可能是同一个目标。

比如：

- 轨迹预测框和当前检测框几乎重合，IoU 可能是 `0.8`
- 两者离得很远，IoU 可能接近 `0`

### 4. 为什么要先做 prediction，再做 association？

因为目标在运动，不能拿上一帧位置生搬硬套到当前帧。

最经典的最小运动模型是常速度模型：

$$ x_{t|t-1} = F x_{t-1|t-1} $$

这里：

- $x_{t-1|t-1}$：上一帧更新后的状态
- $x_{t|t-1}$：当前帧、但还没看当前检测前的预测状态
- $F$：状态转移矩阵

如果直觉化一点，就是：

- 目标上一帧往右移动了 4 个像素
- 那当前帧先猜它还会继续往右移动 4 个像素

这样在遮挡或漏检时也能稍微撑几帧。

### 5. Hungarian matching 为什么老被提？

因为一旦同时有多条轨迹、多个检测，不能每条都独立贪心选最近的，否则很容易“抢框”。

举个例子：

- 有两条历史轨迹 T1、T2
- 当前有两个检测 D1、D2
- T1 对 D1 和 D2 都还算近
- T2 其实只跟 D2 真正接近

如果 T1 先贪心拿走 D2，T2 就可能匹配失败，造成 ID switch。

所以工程里常见做法是：

- 先构造代价矩阵
- 再用 Hungarian algorithm 求全局最优匹配

这篇的 [minimal.py](minimal.py) 为了保持代码短，使用的是 greedy matching。面试时你要主动说明：

> 教学最小版可以用贪心，但生产里常用 Hungarian matching，尤其是目标数多的时候。

### 6. track 生命周期管理为什么重要？

跟踪不是“匹配完就结束”，还得管理轨迹生老病死。

最常见的轨迹状态包括：

- tentative：刚出现，还没完全确认
- confirmed：已经连续匹配到几帧，认为是真目标
- lost / missed：这几帧没匹配上，但先保留
- removed：丢太久了，删除

举个自动驾驶里很常见的例子：

- 前车短暂被大货车挡住 1 帧
- 如果系统立刻删轨迹，下一帧它重新出现时就会变成新 ID
- 这就是典型的 ID switch 风险

所以很多系统都会允许轨迹短时间“失联但不立刻死亡”。

### 7. 最小版跟踪器怎么写？

在 [minimal.py](minimal.py) 里，我保留了一个教学版主线：

- 用 IoU 做关联
- 用上一帧速度做简单预测
- 用 greedy matching 保持代码短
- 用 `missed` 计数做生命周期管理

先看 IoU：

```python
def iou(box_a, box_b):
    inter = inter_w * inter_h
    union = area_a + area_b - inter
    return 0.0 if union <= 1e-6 else inter / union
```

这段很短，但你面试时至少要能说出：

- 交集大，说明更像同一个目标
- 并集把尺度也考虑进来了

再看轨迹更新：

```python
old_center = box_center(self.box)
new_center = box_center(new_box)
self.velocity = new_center - old_center
self.box = new_box
```

这几行在做：

- 取旧框中心
- 取新框中心
- 两者相减，估一个最简单的速度
- 再用新框覆盖旧状态

然后是轨迹丢失处理：

```python
def mark_missed(self):
    self.box = self.predict()
    self.missed += 1
```

这代表：

- 当前帧虽然没检测到它
- 但先用预测框顶一帧
- 同时把丢失计数加一

这就是 online tracking 最核心的“撑一撑”逻辑。

## 面试高频问题

### 1. 检测和跟踪的区别是什么？

- 检测解决“这一帧有什么”
- 跟踪解决“跨帧里谁是谁”

### 2. 为什么多目标跟踪比单帧检测更难？

因为它多了：

- 目标身份延续
- 遮挡恢复
- 漏检容错
- 轨迹管理

### 3. ID switch 是什么？

原本同一个目标的轨迹 ID 在中途变了，或者两个目标的身份被互换。

### 4. IoU matching 的优点和缺点是什么？

- 优点：简单、快、容易实现
- 缺点：遮挡、视角变化、大位移时容易失效

### 5. Kalman filter 在 MOT 里通常干什么？

用来做轨迹状态预测和测量更新，尤其适合处理“位置 + 速度”这类低维连续状态。

### 6. 为什么需要 ReID 特征？

当两个目标位置很近、甚至交叉穿行时，光靠几何位置可能不够，需要外观特征辅助区分。

## 最小实现

完整代码见：[minimal.py](minimal.py)。

这份最小实现包含：

- `iou(...)`
- `Track`
- `greedy_match(...)`
- `SimpleTracker`

测试数据是 4 帧合成框序列，其中第二个目标在第 3 帧短暂漏检：

```python
frames = [
    [[10, 10, 30, 30], [60, 12, 82, 32]],
    [[14, 10, 34, 30], [56, 12, 78, 32]],
    [[18, 10, 38, 30]],
    [[22, 10, 42, 30], [48, 12, 70, 32]],
]
```

这个例子非常适合讲“漏检但轨迹不立即丢”：

- 第 1 个目标一直稳定向右移动
- 第 2 个目标第 3 帧没被检测到
- 但系统不会立刻删掉它
- 第 4 帧重新匹配上后，ID 还能延续

主循环也很值得记：

```python
matches, unmatched_tracks, unmatched_dets = greedy_match(...)
for track_idx, det_idx in matches:
    self.tracks[track_idx].update(detections[det_idx])
for track_idx in unmatched_tracks:
    self.tracks[track_idx].mark_missed()
for det_idx in unmatched_dets:
    self.tracks.append(...)
```

这几段分别在做：

- 先匹配
- 匹配上的更新
- 没匹配上的老轨迹先预测保留
- 没匹配上的新检测创建新轨迹

这就是最小 MOT 的完整闭环。

## 工程关注点

### 1. 检测质量往往决定跟踪上限

漏检和误检太多，再强的跟踪器也很难救。

### 2. 关联指标需要分场景选

车流稠密、行人遮挡、远距离小目标，这些场景适合的代价设计都不完全一样。

### 3. 生命周期阈值要和业务场景匹配

自动驾驶里速度快、遮挡多，阈值太激进容易掉 ID，太保守又会保留很多假轨迹。

### 4. 多传感器跟踪和纯视觉跟踪不是一回事

有雷达或激光雷达时，距离和速度观测会稳很多。

## 常见坑点

### 1. 把跟踪理解成“每帧都重新检测一次”

这只讲到了一半，没讲身份延续。

### 2. 只会说 IoU matching，不会说为什么会出 ID switch

这会显得理解太浅。

### 3. 不区分教学版和生产版

最小实现可以贪心，但工程里更常见的是 Kalman + Hungarian + ReID。

### 4. 忽略轨迹生命周期管理

这在真实场景里是很大的坑。

## 面试时怎么讲

可以按这条线讲：

1. 多目标跟踪的核心不是检测目标，而是让同一个目标在连续帧里保持稳定 ID。
2. 主线通常是检测、运动预测、关联、状态更新、轨迹管理。
3. 最简单的关联指标是 IoU，目标数多时常结合 Hungarian 求全局最优匹配。
4. 遮挡和漏检下，系统通常不会立刻删轨迹，而是先预测保留几帧。
5. 工程里常见框架是 Kalman filter + Hungarian matching，复杂场景再叠加 ReID 特征。
