# 模板 / 泛型 面试攻略

## 这是什么？

这是现代 C++ 里最重要的“零成本抽象”工具之一。

如果只用一句话来讲：

> 模板让你可以把“类型无关的逻辑”写成一份代码，在编译期为不同类型生成具体实现，从而实现泛型编程和编译期多态。

在 AI / C++ 代码里，它非常常见，因为很多逻辑天然会按这些维度变化：

- 数据类型：`float / double / int`
- 累加类型：`float / double`
- 后端或策略：不同 kernel / policy
- 容器或张量视图类型

## 核心机制

### 1. 模板到底解决什么问题？

如果一段逻辑对多种类型都成立，但只因为类型不同就要重复写很多份代码，那就适合用模板。

最典型的例子就是：

- 通用 `dot`
- 通用 `clamp`
- 通用容器
- 通用张量访问器

### 2. 函数模板和类模板的区别是什么？

#### 函数模板

适合：

- 同一类算法逻辑
- 输入类型不同

#### 类模板

适合：

- 整个数据结构或对象都和类型绑定

比如：

- `Tensor<T>`
- `Buffer<T>`
- `Accumulator<T>`

### 3. 模板多态和虚函数多态怎么选？

这也是高频题。

- 模板多态：编译期分发，没有虚函数开销，性能友好
- 虚函数多态：运行时分发，更灵活，适合接口替换

所以很多高性能 AI C++ 代码更偏向模板，而不是在热路径里大量用虚函数。

### 4. AI 代码里，模板常见用在哪？

几个典型方向：

- 同一个 kernel 支持不同 dtype
- 同一个算子支持不同索引类型
- 编译期选择优化策略
- 写通用 tensor / iterator / accessor

要点是：

> 模板最适合“结构相同、类型或策略不同”的场景。

## 面试高频问题

### 1. 模板为什么叫编译期多态？

因为具体实例化发生在编译期，针对不同类型会生成不同版本代码。

### 2. 模板一定更快吗？

不一定，但它通常更容易让编译器内联和优化，也没有虚函数分派开销。

### 3. 模板缺点是什么？

- 编译报错往往更绕
- 二进制体积可能膨胀
- 过度模板化会降低可读性

### 4. AI C++ 里为什么模板特别多？

因为 dtype、index、layout、policy 这些维度都很适合在编译期固定下来，以换取更好的性能。

## 最小实现

下面这个最小例子同时包含函数模板、类模板和模板策略调用：

```cpp
template <typename T>
T dot_product(const std::vector<T>& a, const std::vector<T>& b) {
    if (a.size() != b.size()) {
        throw std::invalid_argument("size mismatch");
    }

    T sum = T{};
    for (std::size_t i = 0; i < a.size(); ++i) {
        sum += a[i] * b[i];
    }
    return sum;
}

template <typename T>
class Tensor1D {
public:
    explicit Tensor1D(std::vector<T> values) : values_(std::move(values)) {}
    const std::vector<T>& values() const { return values_; }

private:
    std::vector<T> values_;
};
```

这里最值得你讲的点是：

- `template <typename T>`：把类型参数抽象出来
- `dot_product(...)`：同一份逻辑可用于 `float`、`double` 等不同类型
- `T sum = T{}`：用值初始化适配泛型类型
- `Tensor1D<T>`：整个类都和类型参数绑定，所以更适合用类模板

配套代码里还补了一个 `apply_inplace`，展示“模板函数接收策略对象”的写法。

## 工程关注点

### 1. 模板适合热路径

如果逻辑简单、调用频繁、类型集合相对明确，模板通常很合适。

### 2. 不是所有接口都该模板化

面向外部模块的稳定 ABI、插件接口、动态加载场景，很多时候还是运行时多态更稳。

### 3. 模板和 `constexpr` / `if constexpr` 经常一起用

这是现代 C++ 里很常见的编译期分支方式。

## 常见坑点

### 1. 过度模板化

容易把简单逻辑写成难读难查的元编程。

### 2. 忘了模板实例化带来的编译和二进制成本

模板不是完全免费的。

### 3. 运行时问题想靠模板解决

模板是编译期工具，不适合处理真正运行时才知道的类型变化。

## 面试时怎么讲

如果面试官问模板，可以按这个顺序讲：

1. 模板是 C++ 的泛型编程工具
2. 它让同一份逻辑在编译期为不同类型生成实现
3. 所以它也常被称为编译期多态
4. AI C++ 里很常用于 dtype、索引类型和策略泛化
5. 相比虚函数，多数情况下模板更适合性能敏感路径

一个简洁版本可以直接讲：

> 模板的核心作用是把类型无关的逻辑写成一份代码，在编译期针对不同类型实例化出具体实现，因此它本质上是一种编译期多态。AI C++ 代码里模板很常见，因为很多算子只是在 dtype、索引类型或策略上不同，逻辑结构本身是相同的。相比虚函数，模板通常更适合性能敏感路径。
