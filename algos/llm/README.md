# LLM 面试攻略

## 这是什么？

这是仓库中负责 LLM 方向内容的总入口。这里不只放 Transformer 基础模块，还要系统整理主流开源 LLM 家族的 paper reading / 技术报告精读。

## 核心机制

LLM 方向的内容分成两层：

- 通用机制：Attention、RoPE、Normalization、KV Cache、MoE、训练稳定性、推理优化
- 模型家族精读：针对主流开源 LLM，以及公开过技术报告但没有正式论文的模型，分别总结其设计取舍和面试讲法

## 面试高频问题

- 某个模型相比前代到底改了什么，收益来自哪里？
- 架构改动是为了训练稳定性、推理吞吐，还是长上下文能力？
- 数据配方、tokenizer、上下文长度、对齐方式分别如何影响结果？
- 同样是开源 LLM，为什么不同家族在推理成本、效果和工程复杂度上差异很大？
- 面试里怎么把一个模型讲成“核心改动 + 设计动机 + 工程代价”的结构？

## 模型家族精读

`paper_reading/` 目录作为模型家族精读索引，维护原则如下：

- 每个主流开源 LLM 家族都要有独立条目
- 如果模型没有正式 paper，但发布了技术报告、blog、系统说明或 model card，仍然要有独立精读条目
- 同一家族的多代模型可以先合并整理；如果某一代改动足够大，再拆成独立文件
- 每个条目至少要覆盖：定位、架构变化、训练配方、上下文长度、对齐方式、推理工程点、典型追问

## 最小实现

本专题的最小实现优先围绕通用机制，而不是复刻整套大模型。例如：

- 手写单头 / 多头 Attention
- 手写 RoPE
- 手写 RMSNorm / LayerNorm
- 最小 KV Cache 推理示例
- 最小 MoE 路由示例

## 工程关注点

- 不同模型家族的 attention 结构、KV cache 组织和并行策略会直接影响推理实现
- 长上下文、MoE、GQA / MQA、推理量化等设计，需要同时从效果和工程代价两边讲清楚
- 面试里不能只背 benchmark，要能解释“为什么这样设计”

## 常见坑点

- 把模型家族精读写成论文摘要，缺少面试表达和工程视角
- 只记结论，不区分哪些收益来自架构，哪些来自数据和训练配方
- 只整理有正式论文的模型，遗漏只有技术报告但面试高频出现的开源模型
- 混淆“模型家族共性”和“某一代版本特有改动”

## 面试时怎么讲

讲某个开源 LLM 时，建议按下面顺序展开：

1. 先说它想解决什么问题，或者相对前代的目标是什么
2. 再说最关键的几处结构或训练改动
3. 接着说这些改动带来的收益和代价
4. 最后补推理部署、长上下文、对齐或 MoE 等工程追问
