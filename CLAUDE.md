# CLAUDE.md

This file provides guidance to Claude Code and other AI coding agents when working with this repository.

> 说明：根目录 `AGENTS.md` 是 `CLAUDE.md` 的软链接，本质上是同一个文件。
> 修改规范时只改其中一份即可，不要把它们当成两份独立文件维护。

## 项目概述

这是一个面向**AI 算法工程岗面试**的 Python 速查仓库。

这里的“算法”主要指：

- Transformer / LLM
- PPO / DPO / RLHF
- CV 基础算法与现代模型
- CUDA / 算子 / 推理优化
- 训练稳定性、分布式训练与部署相关知识

它不是刷题模板库，也不是一个只讲论文结论的笔记仓库，而是一套强调以下目标的内容集合：

- 面试里能讲清楚核心机制
- 高频模块能写出最小可运行实现
- 能从原理、训练、推理、工程四个维度展开
- 能覆盖典型追问和常见误区

## 仓库结构

```text
python-ai-cheatsheet/
├── algos/              # 算法方向专题
│   └── [topic]/        # 未来会逐步填充
│       ├── README.md   # 该专题的面试攻略
│       ├── *.py        # 最小实现、实验脚本、推导验证代码
│       ├── *.cu        # 可选，CUDA 示例
│       └── *.md        # 可选，补充说明
├── tools/              # 可视化、实验、benchmark、辅助脚本
├── README.md           # 项目说明
├── CLAUDE.md           # 本文件
└── AGENTS.md           # 软链接到 CLAUDE.md
```

## 内容定位

新增内容时，默认以以下标准判断是否值得收录：

1. 是否属于 AI 算法岗高频主题或高频追问。
2. 是否能抽象成稳定的知识单元，而不是只适用于某一篇论文。
3. 是否可以配出最小实现、公式推导或工程解释中的至少一种。
4. 是否能明确说清楚适用前提、收益、代价和常见坑点。

如果某个方向“学术上重要但面试中很少单独考”，优先级可以后置。

## 目录规范

当新增一个算法专题时，推荐使用如下结构：

```text
algos/[topic]/
├── README.md           # 必需，先讲原理、追问、实现、工程点
├── minimal.py          # 主推荐最小实现
├── demo_*.py           # 可选，训练/推理/可视化演示
├── kernel_*.cu         # 可选，CUDA 示例
└── notes.md            # 可选，补充记录
```

要求：

- 每个专题目录必须至少有一个 `README.md`
- `README.md` 优先解释“核心机制是什么、为什么这么设计、面试时怎么讲”
- 实现文件名优先使用语义化命名，不要用 `test.py`、`tmp.py` 这类名字
- 如果同一主题有论文版、工程版、最小版，必须明确区分

对于 `algos/llm/` 这一类专题，额外建议使用如下结构：

```text
algos/llm/
├── README.md                  # LLM 总体面试攻略
├── minimal_*.py               # 通用机制最小实现
├── paper_reading/
│   ├── README.md              # 模型家族精读索引
│   └── [family].md            # 某个模型家族的 paper / 技术报告精读
└── notes.md                   # 可选，补充记录
```

## 专题 README 编写要求

每个 `algos/[topic]/README.md` 建议包含以下结构：

```markdown
# [专题名]面试攻略

## 这是什么？
## 核心机制
## 面试高频问题
## 最小实现
## 工程关注点
## 常见坑点
## 面试时怎么讲
## 延伸阅读
```

写作要求：

- 用中文写，尽量直接，不要写成长篇教材
- 优先写高频追问和常见误区，不要只堆概念
- 每个专题都要有一个“最小实现”锚点
- 如果一个结论依赖特定前提，必须写清楚前提
- 代码示例要足够短，能用于现场讲解
- 主体讲解部分优先采用“伴随式代码讲解”写法：关键机制出现时，就在相邻位置给出对应的短代码，并结合代码逐段解释，不要把代码都堆到文末再统一说明
- 讲代码时要明确点出 `__init__` / `forward` / 核心函数里的每一段分别在做什么、为什么这样写
- 代码里应附上必要的讲解性注释，帮助读者把实现和原理对应起来，但不要写成噪声很多的逐行注释
- 如果使用 LaTeX 公式，按 GitHub Markdown 兼容性来写；不要使用 `\operatorname`，优先使用 `\mathrm` 或更基础的写法
- GitHub 块公式必须写成单行形式，也就是直接写成 `$$ 公式 $$`；不要在 `$$` 和公式内容之间、或公式内容内部再额外断成多行，否则 GitHub 页面容易渲染异常

## 根 README.md 维护要求

修改根目录 `README.md` 时，保持以下原则：

- 项目定位始终围绕“AI 算法工程岗面试”
- 列表顺序优先按面试频率排序，其次按追问深度排序
- “计划收录的内容”使用 checklist 形式维护
- 对于已经完成并落库的专题，在对应条目上提供指向该专题 `README.md` 的相对路径链接
- 仓库状态变化时，更新对应说明，不要让 README 与实际结构脱节

## 代码风格要求

代码与内容默认遵守以下原则：

- 命名清晰，公式与实现要能对应起来
- 最小实现优先可读、可讲、可改
- `numpy` / `torch` 可以作为默认实现工具
- 如果涉及 CUDA / Triton，要明确说明优化目标和瓶颈
- 不要为了“代码短”牺牲解释性

## 第三方依赖约定

本仓库允许在 `tools/` 或实验脚本里使用第三方依赖，例如：

- `numpy`
- `matplotlib`
- `torch`

但默认约束如下：

- `algos/` 下的最小实现可以依赖 `numpy` / `torch`
- 如果某段内容依赖 CUDA / Triton / C++ 扩展，必须明确标注环境前提
- 不要在没有必要时引入过重依赖

## AI 协作约定

AI 代理在本仓库工作时，应优先遵守以下原则：

1. 新增内容前，先确认是否符合仓库定位，而不是盲目扩写。
2. 优先补“高频、可解释、可手写最小实现”的内容。
3. 修改 `CLAUDE.md` 时，不要单独再维护一份 `AGENTS.md`。
4. 更新模板或文档时，避免把内容写成教材；更像面试作战手册。
5. 如果某个结论存在适用前提，需要明确写出，不要省略。
6. LLM 方向除了通用机制外，主流开源模型家族要有各自独立的 paper reading 部分；如果模型只有技术报告，也按独立精读条目维护。
7. 新写专题时，主讲解部分默认需要一路伴随代码，一边讲机制一边贴对应短代码，并结合代码解释；不要把主要代码讲解全部堆到“最小实现”一节里。
